Transactional API

Radar does not have a REST API, but it does have a session-less endpoint that allows you to write transactional interections without establishing a session through engine.io. All of these requests are the same message format as that used by the engine.io client and are POST'ed to a single endpoint at /api. The server is capable of recognizing repeated requests from the same identified client under two conditions:

  1. the initial request from the client was to subscribe/sync with a resource
  2. each subsequent request includes the X-RADAR-ID HTTP header with the value specified by the corresponding header in the initial response

Otherwise, each request is treated as if it originates from a separate client.

Status

set a status

curl -H "Content-Type: application/json" -X POST -d '
{
  "op": "set",
  "to":"status:/test/ticket/1",
  "key":"greeting",
  "value":"hello",
  "ack": true
}
' https://localhost:8000/api
Response
{
  "op": "ack",
  "value": true
}

You probably want to set the key to the current user's ID if you want to have each user have it's own value in the same scope.

You can store any arbitrary string as the value.

retrieve a status

curl -H "Content-Type: application/json" -X POST -d '
{
  "op": "get",
  "to":"status:/test/ticket/1"
}
' https://localhost:8000/api
Response
{
  "op": "get",
  "to": "status:/test/ticket/1",
  "value": {
    "greeting": "hello"
  }
}

The keys and values of the hash are determined by the content of the status. Often these are userID: value pairs.

Presence

Presence should only be set from the Radar client, not via the API, as the presence will disappear after the connection has been severed.

retrieve a presence

curl -H "Content-Type: application/json" -X POST -d '
{
  "op": "get",
  "to":"presence:/test/ticket/1",
  "options": { "version": 2 }
}
' https://localhost:8000/api
Response - Presence
{
  "op": "get",
  "to": "presence:/test/ticket/1",
  "value": {
    "1234": {
      "userType": 2,
      "clients": {
        "hdchvkgfj": {}
      }
    },
    "4567": {
      "userType": 0,
      "clients": {
        "jhghjgjkghjk": {},
        "khgfdakjhkmh": { "name": "joe" }
      }
    }
  }
}

The "value" property of the response has a key for each identified userId that has set the presence to "online" and each userId property contains a userType property that corresponds to the value provided by the clients based on configuration properties and a dictionary/hash of client ids that are supplied by each unique socket connection that a userId has used. The value of the client properties is either a dictionary/object representing the value passed in by the client for "userData" when the presence was set to "online".

MessageList

publish a message

curl -H "Content-Type: application/json" -X POST -d '
{
  "op": "publish",
  "to": "message:/test/ticket/1",
  "value": "this is the message",
  "ack": true
}
' https://localhost:8000/api
response
{
  "op": "ack",
  "value": true
}

retrieve a message

curl -H "Content-Type: application/json" -X POST -d '
{
  "op": "sync",
  "to": "message:/test/ticket/1"
}
' https://localhost:8000/api
Response
{
  "op": "sync",
  "to": "message:/test/ticket/1",
  "time": 1407974998784,
  "value": [
    { to: 'message:/test/ticket/1', value: "this is the message" }, 1407974998764,
  ]
}

The response includes all the messages have not yet expired (based on the message retention policy on the server side, e.g. time-limited for some resources).
